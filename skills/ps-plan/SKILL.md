# P는 S이다 — 본질적 문제 정의 프레임워크

기획의 본질은 2형식이다. **P(Problem)는 S(Solution)이다.**
S를 만들어내는 것이 아니라, 진짜 P를 찾으면 S는 이미 그 안에 있다.

## 절대 원칙

- 눈앞의 문제를 즉시 P로 확정하지 않는다
- S(해결)부터 생각하지 않는다
- P가 확정되기 전에 실행하지 않는다

## 단계별 프로토콜

### 1단계: P 의심 — 이것이 진짜 P인가?

요청이나 문제를 받으면, 곧바로 해결에 들어가지 않고 다음을 수행한다:

1. 표면에 드러난 문제를 적는다 (표면 P)
2. 스스로 질문한다:
   - "이것은 원인인가, 결과(증상)인가?"
   - "이 아래에 더 근원적인 문제가 있지 않은가?"
   - "이 P를 해결해도 진짜 문제는 남아 있지 않은가?"
3. 최소 2~3층을 파고들어 더 본질적인 P를 탐색한다

```
표면 P: (사용자가 말한 문제)
  ↓ 왜?
중간 P: (한 층 더 깊은 원인)
  ↓ 왜?
본질 P: (이것을 해결하면 위의 모든 것이 해소되는 근원)
```

### 2단계: P 확정 — 한 문장으로 선언

- 본질적 P를 한 문장으로 정리한다
- 검증: "이 P를 해결하면 표면의 증상들도 함께 해소되는가?"
- 해소되지 않는다면 → 1단계로 돌아간다
- P는 반드시 원인이어야 하며, 결과여서는 안 된다

### 3단계: S 도출 — P 안에서 발견한다

- P가 정확히 확정되면, S는 P 안에서 자연스럽게 드러난다
- S를 "발명"하지 않는다. P에서 "발견"한다
- "P는 S이다" 등식을 한 문장으로 제시한다

```
P: [본질적 문제]
S: [P로부터 도출된 해결]
P는 S이다: [한 문장 등식]
```

### 4단계: 선언 후 실행

작업을 시작하기 전에, 반드시 사용자에게 다음을 보여준다:

```
[P] 본질적 문제: ...
[S] 해결 방향: ...
```

- 사용자가 P에 동의하면 실행으로 넘어간다
- 사용자가 P에 이의를 제기하면 1단계로 돌아간다
- S가 여러 작업으로 나뉘거나 병렬/팀 실행이 필요하면 → ps-task를 적용하여 task를 구성한다

## 적용 범위

이 프레임워크는 다음 상황에서 반드시 적용한다:

- 새로운 기능 기획/설계
- 버그 분석 및 디버깅
- 아키텍처 의사결정
- 리팩토링 방향 결정
- 전략적 판단이 필요한 모든 요청
- "왜?"라는 질문이 필요한 모든 맥락

## 경계

- 단순 조회, 파일 읽기, 사실 확인 등 명확한 작업에는 적용하지 않는다
- P 탐색이 과도한 지연을 유발하지 않도록, 단순한 요청은 빠르게 처리한다
- 핵심은 "생각 없이 바로 실행하지 않는 것"이지, "모든 것을 느리게 하는 것"이 아니다
